# 学習ロードマップ活用レポート

## 活かせる内容
- ルーティング→コントローラ/サービス→ビュー/レスポンスで原因を分解するデバッグ手法
- 例外を意図的に起こし状態を観察する習慣（`raise`/`throw`/`debugger`）
- インタラクティブ環境でのデータ操作（Rails console ⇔ node/ts-node + Sequelize）
- Docker 環境での依存追加・再起動・ログ確認フロー
- `nil`/`undefined` 起点のエラー検知と防御的実装（スキーマ/型・オプショナルチェーン）
- 典型エラーのパターン学習（404・ハンドラ未登録・テンプレート欠落・`undefined` 参照）

## 活かし方（プロジェクト具体例）
- 新規 API 追加時に「ルート定義→サービス実装→DB操作→レスポンス」を段階確認し、各段で `debugger` と簡易レスポンスを挟んで状態可視化
- Node REPL/ts-node でシードデータを先に作り、API→フロントの流れを再現テストしてから UI 調整
- 404/500 の発生箇所を Supertest で自動化し、回帰テストに組み込む
- `undefined` 参照防止に Zod/型ガードを導入し、フロントの props 受け渡しにオプショナルチェーンを徹底
- Docker コンテナで依存追加後は再ビルド→起動→ログ確認の手順を標準運用として README に追記

## 学習ロードマップ

| フェーズ | 期間目安 | 重点トピック | 実践タスク |
| --- | --- | --- | --- |
| 1. 基礎デバッグ思考 | 1週 | エラーメッセージ読解、ルーティング→ビューの流れ | Express で意図的に 404/500 を起こし、原因階層をメモ |
| 2. インタラクティブ検証 | 1週 | node/ts-node REPL, Sequelize 基本 CRUD | REPL で User/Workout を作成→API 経由で取得 |
| 3. ブレークポイント活用 | 1週 | `node --inspect`, VSCode debug, `debugger` | 1 API エンドポイントでリクエスト中の state を追跡 |
| 4. エラー防御設計 | 1週 | スキーマバリデーション(Zod), 型活用, オプショナルチェーン | リクエスト DTO を Zod で検証し、`undefined` 起因のバグを 0 件に |
| 5. 自動テスト導入 | 1–2週 | Supertest/Vitest 基本、失敗先行テスト | 代表 404/500/成功ケースの API テストを追加し CI に組込み |
| 6. 運用とログ | 1週 | Docker 再ビルド手順、構造化ログ/エラートラッキング | 依存追加→`docker-compose up --build`→ログで確認する手順をドキュメント化 |
| 7. 仕上げ | 1週 | リグレッション防止、コードレビュー観点 | 直近のバグをケース化し、デバッグ手順チェックリストを作成 |

## 一言での効果
階層的デバッグ思考と防御的設計・自動テストが身につき、API/フロント双方のバグ検知が初動で決まり、回帰リスクを最小化できるチームになる。
